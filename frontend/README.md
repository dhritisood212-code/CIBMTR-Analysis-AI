# `frontend/` — React artifact viewer (BUILT)

A Vite + React + TypeScript app that renders the reproduction flow and the artifact bundle.
It type-checks clean (`tsc --noEmit`) and builds (`vite build`). The sections below document
the view contract it implements.

## Run it

```bash
cd frontend
npm install
npm run dev        # http://localhost:5173  (proxies /studies,/runs → backend :8000)
npm run build      # production bundle to dist/
npm run typecheck
```

**Demo mode:** if the backend isn't reachable on load, the app falls back to demo mode and
renders the real fixtures generated by `examples/P-5297-synthetic/` (the passing match report
with color-coded classes, the golden `reproduce_P-5297.R`, the bundle list) — so the viewer is
usable with no backend running. Point it at a live backend and it uses the API instead.

## Screens

1. **Study catalog** — `GET /studies`. Cards showing metadata + links only. Each card links to
   the CIBMTR manuscript, dataset zip, and data dictionary. **No dataset is hosted here.**
2. **Session setup / compliance gate** — before a run:
   - a link to the dataset + a required **"I downloaded this from CIBMTR and accept its Terms &
     Conditions"** checkbox (maps to `tc_confirmed`),
   - a session-scoped upload for the dataset + dictionary (ephemeral; deleted on session end),
   - a plain disclaimer: research/educational tool, not medical advice, CIBMTR did not review
     or endorse the analysis.
3. **Run progress** — `POST /runs` then poll `GET /runs/{id}`. Renders the stage timeline from
   `RunState.events` (interpret → cohort → analyze → compare → diagnose → done).
4. **Artifact viewer** — the core screen. Fetches artifacts via
   `GET /runs/{id}/artifacts/{name}` and renders:
   - **Match report** with reproducibility classes **color-coded**
     (`exact`=green, `within-tolerance`=teal, `coarsening-limited`=amber, `not-reproducible`=grey),
     Table 1 comparison first, then per-target expected vs observed + reason.
   - The **`reproduce_<study>.R` script** with syntax highlighting (read-only; the point is it's
     inspectable).
   - **Downloadable bundle** (script + results + match-report + renv.lock/sessionInfo).

## Color-coding contract

The match report's YAML front matter (`scores[].class`, `scores[].verdict`) drives the colors,
so the UI reads the machine-readable classes rather than parsing prose. A `fail` verdict with
an all-`coarsening-limited` explanation should read visually as "interpretable finding," not a
red X — this is a product requirement, not cosmetic.

## New-study mode

Same screens, but the session-setup form accepts a user analysis plan (`new_study_plan`) and no
published target; the artifact viewer omits the match report and shows the drafted analysis for
expert review.

## Suggested stack

Next.js + TypeScript, `react-markdown` for the reports, `shiki`/Prism for R highlighting,
polling via `swr`. No secrets in the client; all model calls happen server-side.
